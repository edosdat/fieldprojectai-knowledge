F01 — Philosophie & structure d'un projet

F01 — Philosophie & positionnement Bulldozair

Catégorie : Fondamentaux | Usage : Q&A · Onboarding

Ce que Bulldozair est — et n'est pas

Bulldozair n'est pas un outil libre ou un bloc-notes numérique. C'est un outil de procédure, de suivi, de preuve et de reporting. Tout dans Bulldozair doit être pensé pour produire une donnée exploitable.

Les 3 piliers

➜ Tout projet Bulldozair doit être conçu à partir du reporting final.

Pourquoi cette philosophie ?

Les données terrain ne valent que si elles sont exploitables.

Un projet sans structure = données mortes.

Un projet bien structuré = rapport produit en quelques clics.

⚠ Bulldozair mal configuré donne l'illusion du travail sans valeur ajoutée réelle.

PilierDescription
La tâcheUnité minimale de donnée. 1 tâche = 1 élément unique (observation, réserve, FNC, point...).
Le workflowCe qui fait évoluer la tâche dans le temps. Manuel, collaboratif, structuré.
Le reportingLa finalité du projet. Ce que l'on veut sortir conditionne toute la structure.

F03 — Limites techniques Bulldozair

Limites réelles observées sur le terrain

Ces limites ne sont pas officiellement documentées, mais sont constatées sur des centaines de projets. Les ignorer entraîne des problèmes de performance et d'usage.

Que faire si on approche des limites ?

Tâches > 5 000

Découper par bâtiment, gare, tronçon ou phase

Créer un espace par lot ou par zone

Archiver les tâches clôturées

Utilisateurs > 20

Créer des espaces séparés par équipe ou lot

Utiliser des groupes pour limiter la visibilité

Séparer MOE/MOA/Entreprises dans des espaces distincts

➜ En cas de doute sur la volumétrie prévisionnelle : prévoir le split dès le départ.

Tâches par espace5 000 maxRalentissements, rapports lourds, complexité ingérable

F04 — Attribution vs Partage

F04 — Attribution vs Partage

Catégorie : Fondamentaux | Usage : Configuration · Formation

La métaphore de l'email

Règles d'usage

Attribution → une personne ou un groupe responsable du traitement

Partage → les parties prenantes qui doivent suivre sans agir

Les deux ensemble → détermine qui voit quoi dans le projet

Mécanique de visibilité complète

⚠ Un utilisateur non attribué et non en partage ne voit pas la tâche — même s'il est dans l'espace.

➜ Toujours partager par défaut à un groupe interne incluant l'admin pour éviter les tâches invisibles.

F05b — Groupes Bulldozair

F05b — Les groupes Bulldozair

Catégorie : Fondamentaux | Usage : Configuration · Setup · Partage

Pourquoi les groupes sont essentiels

Dans Bulldozair, l'attribution et le partage se font soit sur une personne, soit sur un groupe. Les groupes permettent d'attribuer une tâche à 'Entreprise GO' sans devoir sélectionner chaque ouvrier individuellement.

➜ Sans groupes bien configurés, les attributions sont individuelles → surcharge manuelle → erreurs → tâches invisibles.

Les deux types de groupes à créer

Questions à poser lors du setup des groupes

Voulez-vous créer un groupe interne MOE ? (recommandé — sera utilisé pour le partage par défaut)

Quelles entreprises / lots ont plusieurs personnes ? (créer un groupe par lot)

Comment s'appellent vos lots ? (noms exacts pour les groupes)

Format de collecte — ce que l'agent doit demander

Pour chaque groupe, collecter :

Groupes recommandés par cas d'usage

Lien groupes ↔ partage par défaut

Le partage par défaut doit TOUJOURS pointer vers le groupe interne MOE — pas vers des individus. Si ce groupe n'est pas créé, les tâches créées sans attribution manuelle ne seront vues que par leur créateur.

⚠ Pas de groupe interne = pas de partage par défaut fiable = tâches invisibles dans les rapports (Anti-pattern #9, F25).

➜ Créer le groupe interne MOE EN PREMIER, avant d'inviter les autres utilisateurs.

F07 — Tags — familles officielles

F07 — Tags — Familles officielles par cas d'usage

Catégorie : Composants | Usage : Configuration · Reporting

Principe général — recommandations vs obligations

INSTRUCTION AGENT : les familles de tags ci-dessous sont des RECOMMANDATIONS FORTES de la Méthode Éloi — pas des lois immuables. Le client peut utiliser d'autres noms de familles s'il a une raison valable. L'agent doit proposer les familles officielles en premier, expliquer pourquoi elles sont meilleures, mais accepter les choix du client si celui-ci insiste.

➜ L'agent propose, explique, conseille — mais ne bloque jamais un choix du client sauf si c'est marqué INTERDIT.

Familles officielles par cas d'usage

Suivi de chantier

✅ RECOMMANDÉ — SUIVI DE CHANTIER

Ces familles sont la Méthode Éloi — fortement conseillées. Autres noms acceptés si le client a une bonne raison.

FAM_01 = "Nature" ex: Visite, Contrôle, Sécurité, Observation

FAM_02 = "Zone" ex: RDC, R+1, Façade Est, Bâtiment A

FAM_03 = "Activité" ex: CFO, CFA, GO, VRD, Plomberie

OPR

✅ RECOMMANDÉ — OPR

Ces familles sont la Méthode Éloi — fortement conseillées. Autres noms acceptés si le client a une bonne raison.

FAM_01 = "Zone" ex: RDC, R+1, Bâtiment A, Appartement 12

FAM_02 = "Activité" ex: CFO, CFA, GO, Plomberie, Peinture

❌ INTERDIT : Famille "Nature" en OPR — tout est réserve — cette famille n'apporte aucune valeur et surcharge les rapports

Si le client veut une autre famille (ex : "Catégorie", "Phase", "Criticité") → c'est DÉCONSEILLÉ mais possible. Signaler le risque sur le reporting puis accepter.

FNC

✅ RECOMMANDÉ — FNC

Ces familles sont la Méthode Éloi — fortement conseillées. Autres noms acceptés si le client a une bonne raison.

FAM_01 = "Type FNC" ex: Qualité, Sécurité, Documentaire, Délai

FAM_02 = "Zone" ex: Tronçon A, Zone 1, Bâtiment C

FAM_03 = "Activité" ex: GO, CFA, CFO, VRD

FAM_04 = "Criticité (opt.)" ex: Mineure, Majeure, Bloquante

CRR

✅ RECOMMANDÉ — CRR

Ces familles sont la Méthode Éloi — fortement conseillées. Autres noms acceptés si le client a une bonne raison.

FAM_01 = "Section" ex: Administratif, Sécurité, Avancement, OS, MOA

FAM_02 = "Sous-section (opt.)" ex: À définir selon le format du client

QSE / Audit

✅ RECOMMANDÉ — QSE / AUDIT

Ces familles sont la Méthode Éloi — fortement conseillées. Autres noms acceptés si le client a une bonne raison.

FAM_01 = "Nature QSE" ex: Sécurité, Qualité, Environnement

FAM_02 = "Zone" ex: Zone chantier, Atelier, Extérieur

FAM_03 = "Gravité (opt.)" ex: Faible, Significatif, Critique

Règles absolues sur les tags — celles-ci sont des INTERDITS réels

Contrairement aux noms de familles (recommandations), ces règles sont des interdits structurels.

❌ INTERDIT : 0 tag dans un projet — sans tags, les rapports sont vides et le projet inexploitable (Anti-pattern #1, F22)

❌ INTERDIT : Nom d'une entreprise comme famille d'Activité — les entreprises changent, les activités restent — crée une incohérence structurelle

❌ INTERDIT : Mélanger deux logiques métier dans un même espace de tags — voir F02 — 1 espace = 1 cas d'usage

F09 — Formulaires

F09 — Les formulaires

Catégorie : Composants | Usage : Configuration · Formation

Formulaires : quand les utiliser ?

Un formulaire sert à structurer le contenu d'une tâche, imposer des champs obligatoires, standardiser une checklist et extraire des données précises. Il est attaché à la tâche, pas un objet autonome. Il est lié au compte, pas au projet.

Quand utiliser un formulaire ?

Checklist normée (QSE, sécurité, qualité)

Plan de contrôle technique (armatures, coffrage, étanchéité)

Audit d'équipement

Inspection structurée avec exigence normative

Besoin de champs obligatoires pour la saisie

Quand NE PAS utiliser un formulaire ?

OPR — saisie trop lente, nuit à la productivité

CRR — les tâches structurent déjà l'information

Observations simples — surcharge inutile

Situations à forte volumétrie (OPR, suivi)

Règles de construction

Rester simple : moins de 20 champs

Diviser en plusieurs formulaires si besoin (plutôt qu'un de 80 champs)

Penser l'export Excel/CSV dès la conception

Limiter les champs obligatoires à l'essentiel

Tester le formulaire sur le terrain avant déploiement

➜ Plus un formulaire est simple, meilleur est son usage terrain.

CORRESPONDANCES TERMINOLOGIQUES — INSTRUCTION AGENT

Ces termes sont des synonymes exacts dans Bulldozair. Une question avec l'un de ces mots = répondre avec cette fiche.

ALIASES "Modèle 3D" : IFC = BIM = modèle numérique = maquette 3D = fichier 3D = modèle BIM = 3D → même chose, même réponse.

ALIASES "Plan 2D" : PDF plan = plan PDF = fond de plan = plan tuilé = plan d'étage = plan de masse = plan DWG exporté PDF → même chose, même réponse.

ALIASES "Fond de carte" : SIG = GeoJSON = carte géographique = géolocalisation = fond cartographique = carte satellite = plan géoréférencé → même chose, même réponse.

Les 3 types de supports dans Bulldozair

Bulldozair supporte 3 types de supports : les plans 2D (PDF), les fonds de carte SIG (GeoJSON), et les modèles 3D (IFC/BIM). Ils sont superposables dans le même espace.

Peut-on charger des modèles 3D dans Bulldozair ?

✅ OUI — Bulldozair supporte nativement les fichiers IFC (modèles 3D / BIM). On peut charger un modèle 3D, naviguer dedans sur tablette ou desktop, et épingler des tâches directement sur le modèle.

Réponse complète à donner à l'utilisateur :

"Oui. Bulldozair supporte les fichiers IFC (modèles 3D / BIM). Tu peux charger ton modèle, naviguer dedans sur tablette, et épingler des tâches directement sur la maquette. Format requis : IFC. Les plans 2D PDF et le modèle 3D sont superposables dans le même espace."

Anti-pattern #1 — Absence de tags

C'est la cause n°1 d'échec dans Bulldozair. Un projet sans tags est inexploitable.

Symptômes

Impossible de filtrer les tâches

Rapports illisibles ou vides

Aucune segmentation par zone, activité ou nature

Données plates — tout est au même niveau

Projet progressivement abandonné par les équipes

Pourquoi cela arrive ?

L'utilisateur ne sait pas ce qu'est un tag

Absence de template projet avec tags pré-créés

Confusion entre tags, statuts et formulaires

Setup fait sans accompagnement

Solution Méthode Éloi

Définir les tags APRÈS avoir vu le rapport attendu

Imposer au minimum 2 familles, idéalement 3

Structurer : Nature / Zone / Activité

Créer un template avec tags pré-créés pour chaque cas d'usage

Former les utilisateurs sur le rôle des tags dans le reporting

F23

F23 — Anti-pattern : Tâches fourre-tout

F23 — Anti-pattern #2 : Tâches fourre-tout

Catégorie : Anti-patterns | Usage : Q&A · Diagnostic

Anti-pattern #2 — Tâches fourre-tout

Deuxième erreur la plus courante. Une tâche regroupant plusieurs observations est inutilisable pour le reporting et le workflow.

Symptômes

40 photos dans une seule tâche

20 problèmes différents mélangés

Impossible de trier, filtrer ou attribuer précisément

Workflow inutilisable (qu'est-ce qui est « Terminé » ?)

Rapports avec 1 ligne = 50 éléments

Pourquoi cela arrive ?

Confusion entre Bulldozair et WhatsApp / messagerie

Utilisateurs pressés, sans formation

Mauvaise compréhension du concept de granularité

Solution

➜ 1 tâche = 1 observation = 1 responsable = 1 statut = 1 ligne dans le rapport.

En pratique : lors d'une visite, créer autant de tâches que d'observations. Même si cela semble long, c'est la condition pour que le reporting soit exploitable.

✔ Message agent : « Plusieurs observations distinctes semblent regroupées dans cette tâche. La bonne pratique est de créer une tâche par observation. Souhaitez-vous de l'aide pour restructurer ? »

F24 — Anti-pattern : Statuts & workflow

F24 — Anti-patterns #3 à #7 : Statuts & workflow

Catégorie : Anti-patterns | Usage : Q&A · Diagnostic

Anti-patterns liés aux statuts et au workflow

AP #3 — Pas de statuts (ou statuts non utilisés)

Symptômes : 100 % des tâches en statut initial, aucun avancement lisible

Cause : workflow mal configuré, trop complexe, ou absence de formation

Solution : 4 à 6 statuts simples, adaptés au cas d'usage

AP #4 — Absence d'attribution

Symptômes : aucune responsabilité visible, les entreprises ne voient rien

Cause : méconnaissance de la différence attribution / partage

Solution : attribution systématique dès la création, groupes d'entreprises

AP #5 — Espace trop large

Symptômes : > 5 000 tâches, > 20 utilisateurs, lenteurs, bruit

Cause : pas de découpage anticipé

Solution : split par zone, lot, bâtiment ou cas d'usage

AP #6 — Workflow trop complexe

Symptômes : 12 à 16 statuts, transitions incompréhensibles, workflow jamais utilisé

Cause : sur-ingénierie du processus

Solution : max 6 statuts, pensés pour le terrain pas pour l'organigramme

AP #7 — Mauvaise gestion des plans

Symptômes : plans remplacés non superposables, pins décalés

Cause : méconnaissance du processus de mise à jour des plans

Solution : vérifier la superposabilité AVANT tout remplacement

F25 — Anti-pattern : Utilisateurs & formulaires

F25 — Anti-patterns #8 à #11 : Utilisateurs & formulaires

Catégorie : Anti-patterns | Usage : Q&A · Diagnostic

Anti-patterns : utilisateurs, partage, formulaires, template

AP #8 — Trop d'utilisateurs dans un espace

Symptômes : spam de notifications, surcharge visuelle, dérive du projet

Solution : 20 max, groupes internes, espaces séparés

AP #9 — Pas de partage par défaut

Symptômes : tâches invisibles pour le MOE, reporting incomplet

Cause : oubli de configuration ou méconnaissance

Solution : partage par défaut sur un groupe interne incluant l'admin

AP #10 — Formulaires trop longs

Symptômes : utilisateurs ne remplissent pas, données incomplètes, rapports creux

Cause : sur-engineering du formulaire

Solution : plusieurs petits formulaires > un énorme. Limiter les champs obligatoires.

AP #11 — Pas de template projet

Symptômes : tags incohérents entre projets, workflows différents d'un site à l'autre

Cause : chaque projet recréé de zéro

Solution : un template validé par cas d'usage, déployé systématiquement

➜ Les anti-patterns 8 à 11 sont tous évitables avec un bon setup initial et un template validé.
